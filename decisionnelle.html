J'ai conçu et implémenté un entrepôt de données — une base pensée non pas pour stocker, mais pour analyser. L'objectif : permettre à une entreprise de répondre à ses questions stratégiques en quelques secondes.

Le contexte

Pas une base classique — une base pour décider

🏭
Une base classique répond à "quoi"
Elle stocke les faits : une commande passée, un client enregistré, un produit vendu. Elle est pensée pour les opérations du quotidien.
🔭
Un entrepôt répond à "pourquoi"
Il agrège, croise, compare. Qui vend le plus ? Dans quelle région ? Sur quelle période ? C'est la base qui alimente les décisions stratégiques.
⚡
Deux moteurs, deux usages
J'ai travaillé avec PostgreSQL pour la modélisation relationnelle, et ClickHouse pour les requêtes analytiques sur de gros volumes — un moteur orienté colonnes, bien plus rapide pour les agrégations.

Les analyses

Trois façons d'explorer les mêmes données

Zoomer sur une période
On peut descendre du global vers le détail — regarder les ventes de l'année, puis du trimestre, puis du mois — sans changer de base.
Année → Trimestre → Mois → Jour GROUP BY
Croiser les dimensions
Qui vend quoi, où et quand ? On croise vendeur × produit × région × période pour faire apparaître les tendances cachées.
Vendeur × Produit Région × Période ROLLUP
Calculer des tendances
Calculer la moyenne glissante du chiffre d'affaires sur deux ans pour lisser les variations saisonnières et voir l'évolution réelle.
Moyenne glissante WINDOW FUNCTION OVER()
Le code

Ce que j'ai écrit

-- Table centrale : les faits de vente
CREATE TABLE fait_vente (
  id_vente     SERIAL PRIMARY KEY,
  id_temps     INT REFERENCES dim_temps(id_temps),
  id_produit   INT REFERENCES dim_produit(id_produit),
  id_vendeur   INT REFERENCES dim_vendeur(id_vendeur),
  id_lieu      INT REFERENCES dim_lieu(id_lieu),
  chiffre_affaires  DECIMAL(12,2),
  quantite          INT,
  plus_value        DECIMAL(12,2)
);

-- Dimension Temps avec hiérarchie année → mois → jour
CREATE TABLE dim_temps (
  id_temps   SERIAL PRIMARY KEY,
  date_vente DATE,
  jour       INT,
  mois       INT,
  trimestre  INT,
  annee      INT
);

-- Dimension Lieu avec hiérarchie ville → région → pays
CREATE TABLE dim_lieu (
  id_lieu    SERIAL PRIMARY KEY,
  ville      VARCHAR(100),
  region     VARCHAR(100),
  pays       VARCHAR(100)
);
-- Chiffre d'affaires par vendeur et trimestre
SELECT
  v.nom_vendeur,
  t.annee,
  t.trimestre,
  SUM(f.chiffre_affaires) AS ca_total,
  SUM(f.quantite)         AS nb_ventes,
  ROUND(AVG(f.plus_value), 2)  AS plus_value_moy
FROM fait_vente f
JOIN dim_vendeur v  ON f.id_vendeur = v.id_vendeur
JOIN dim_temps   t  ON f.id_temps   = t.id_temps
GROUP BY ROLLUP(v.nom_vendeur, t.annee, t.trimestre)
ORDER BY ca_total DESC;

-- Qui réalise les meilleures plus-values ?
SELECT
  v.nom_vendeur,
  SUM(f.plus_value) AS pv_totale,
  RANK() OVER (ORDER BY SUM(f.plus_value) DESC) AS classement
FROM fait_vente f
JOIN dim_vendeur v ON f.id_vendeur = v.id_vendeur
GROUP BY v.nom_vendeur;
-- Moyenne glissante du CA sur 2 ans (marché locatif)
SELECT
  t.annee,
  t.mois,
  SUM(f.chiffre_affaires) AS ca_mensuel,
  AVG(SUM(f.chiffre_affaires)) OVER (
    ORDER BY t.annee, t.mois
    ROWS BETWEEN 23 PRECEDING AND CURRENT ROW
  ) AS moy_glissante_24mois
FROM fait_vente f
JOIN dim_temps t ON f.id_temps = t.id_temps
GROUP BY t.annee, t.mois
ORDER BY t.annee, t.mois;
Ce que j'en retiens

3 choses que ce projet m'a apprises

On part toujours du besoin
Avant de créer une seule table, j'ai dû répondre à "quelles questions veut-on poser ?". La structure technique découle de ça, pas l'inverse.
La granularité change tout
Stocker par jour permet d'agréger au mois et à l'année. Stocker par mois interdit de descendre au jour. Le choix du niveau de détail est irréversible.
ClickHouse, un autre monde
Sur des millions de lignes, ClickHouse répond en millisecondes là où PostgreSQL met plusieurs secondes. Les bases orientées colonnes sont faites pour ça.
